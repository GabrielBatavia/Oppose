# Oppose Reference Context for AI Coding Agent

File purpose:
This document explains how to use the manually generated Oppose reference pages and illustration assets while rebuilding or implementing the Oppose mobile app.

Recommended location in repo:
docs/AI_AGENT_REFERENCE_CONTEXT.txt

Reference folders:
- docs/refrences/pages
- docs/refrences/assets

Important spelling note:
The current folder name is `refrences`, not `references`. Use the existing path exactly unless the project owner renames it.

===============================================================================
1. PRODUCT CONTEXT
===============================================================================

Oppose is a daily social AI debate app.

The product should feel like:
- a warm daily social app
- a friendly debate space
- a comfortable place for friends to discuss different takes
- AI-assisted, but always user-controlled
- playful, respectful, and emotionally safe

The product should NOT feel like:
- a political conflict platform
- a casino or prediction/betting app
- a corporate AI dashboard
- a cold productivity tool
- a futuristic control panel
- a generic SaaS template

Core product idea:
Users can talk with friends, create debate rooms, chat, invite AI as a helper, and receive optional summaries. The app rewards respectful conversation, curiosity, and open-minded disagreement.

Core brand line:
“Different take, better talk.”

===============================================================================
2. HOW TO USE THE REFERENCE PAGES
===============================================================================

Reference pages folder:
docs/refrences/pages

The reference page images are the primary visual and UX direction for the app.

Use them as:
- layout references
- spacing references
- component hierarchy references
- color/texture references
- visual mood references
- screen-specific UX references

Do NOT use them as:
- static background screenshots
- full-screen image replacements for real UI
- final text rendering assets
- hardcoded bitmap layouts

The app must be built with real native/components:
- real buttons
- real text
- real inputs
- real navigation
- real scroll views
- real toggles
- real chat cells
- real cards
- real accessibility labels
- real responsive layout

The reference pages show how the UI should look, but the implemented app must remain flexible, responsive, and maintainable.

If a visual detail appears in the page reference but is hard to implement exactly, prioritize:
1. UX clarity
2. component correctness
3. visual consistency
4. performance
5. exact decorative detail

Do not sacrifice usability for decoration.

===============================================================================
3. HOW TO USE THE ASSET FOLDER
===============================================================================

Assets folder:
docs/refrences/assets

The asset images are manually generated illustration assets for the app.

Use assets for:
- Bima mascot illustrations
- user avatar illustrations
- AI robot / AI Bima illustrations
- badges
- decorative stickers
- paper/tape textures
- custom icons that define the Oppose brand
- room/topic decorative elements
- profile badge visuals
- onboarding and safety illustrations

Do NOT bake important UI text into image assets.
All important interface copy should be rendered as real app text.

Correct usage:
- Use a Bima illustration as an Image component.
- Use a badge PNG as an Image component.
- Use a tape/sticker PNG as decorative Image.
- Use native Text for titles, labels, button text, field labels, and accessibility.

Incorrect usage:
- Importing an entire generated screen as a background.
- Cropping important UI from a screenshot.
- Using image text for titles or forms.
- Building the app as static images.

Recommended asset approach:
- Keep assets organized by page or semantic use.
- Use transparent PNGs where available.
- Use 2x or 3x density if the project supports it.
- Optimize large assets before production.
- Prefer reusable assets over duplicated page-specific crops.
- Maintain meaningful names.

Recommended naming pattern:
oppose_[page]_[asset_description].png

Examples:
oppose_live_room_avatar_ai_bima.png
oppose_profile_badge_respect_champion_crown.png
oppose_ai_consent_summary_delete_icon.png

4. BRAND STYLE SYSTEM
===============================================================================

Oppose visual style:
- warm handmade scrapbook
- cream paper background
- torn paper cards
- sticker-like illustration assets
- soft shadows
- rounded cards
- playful but clean layout
- Javanese-inspired details
- friendly mascot presence
- hand-drawn accent marks

Core colors:

- Mint green: positive, safe, selected state, social warmth
- Sunflower yellow: highlights, tape, small emphasis stickers
- Indigo blue: secondary accent, AI/translation/special detail
- Warm cream: main background and paper cards
- Soft gray/black: body text and neutral icons

- Maroon Red: #8E1D2C

- Indigo Blue: #384BA7
- Warm Cream: #FFF6E6
- Soft Ink: #222222
- Muted Gray: #6F6F6F

Typography direction:
- Friendly rounded heading type
- Highly readable body type
- Slightly playful but not childish
- Avoid overly corporate fonts
- Avoid futuristic sci-fi fonts

Layout direction:
- large friendly headings
- generous vertical spacing
- rounded paper cards
- clear action hierarchy
- one obvious primary action per screen
- bottom navigation for main app screens
- soft card shadows
- decorative assets placed around edges, never blocking usability

===============================================================================
5. GLOBAL UX RULES
===============================================================================

1. Build a real social app, not an art board.
The generated references are visually rich, but the app must be usable.

2. Important controls must be obvious.
Especially:
- Join room
- Leave room
- Mute
- Ask AI
- Turn off AI
- Report
- Block
- Save/delete summary

3. AI must always feel controlled by the user.
AI states should be visible:
- AI Off
- Not listening yet
- Listening
- Thinking
- Speaking
- Summary generated
- Memory off

4. AI should not look like a human user.
AI avatars and messages must be clearly labeled as AI.

5. Safety features must be visible and calm.
Report/block/leave actions should be easy to find, but not visually terrifying.

6. Use native text for all UI copy.
The generated images can guide typography and layout, but final UI text must be coded.

7. Avoid overcrowding.
The visual system is decorative, but core UX should remain simple.

8. Keep page backgrounds light and warm.
Most Oppose screens use warm cream backgrounds with paper texture.

9. Preserve the mascot identity.
Bima is a major brand asset and should appear especially in onboarding, home, AI, profile, and summary moments.

10. Respect mobile constraints.
Make all key actions reachable and readable on normal smartphone sizes.

10. RESPONSIVE AND ACCESSIBILITY RULES
===============================================================================

Responsive:
- Support common iPhone and Android dimensions.
- Avoid hardcoded absolute positioning for all layout.
- Use safe-area padding.
- Use scroll views for long screens.
- Keep bottom buttons reachable.
- Avoid clipping on smaller devices.

Accessibility:
- All buttons need accessible labels.
- AI state must be text-readable, not color-only.
- Selected states must use checkmarks or labels, not color alone.
- Text contrast must remain readable over cream backgrounds.
- Important text should be native text.
- Tap targets should be comfortable.

Performance:
- Optimize PNG assets.
- Avoid loading giant full-resolution images for small stickers.
- Cache frequently used assets.
- Use vector/native icons for simple system icons when possible.
- Use PNG illustrations only for brand-specific assets.

- AI mode selection
- Summary privacy setting
- Delete summary

AI safety copy:
- “AI only listens when it is turned on.”
- “You can mute or remove AI anytime.”
- “Summaries are optional and deletable.”

Never imply hidden listening.
Never hide the AI off control.
Never make AI look like a human user.
Never save summaries without a visible user-facing privacy state.

===============================================================================
12. DESIGN DO AND DO NOT
===============================================================================

DO:
- Use warm cream paper backgrounds.
- Use maroon for primary actions.
- Use mint green for selected/safe/success states.
- Use yellow for highlight stickers and tape.
- Use indigo sparingly for secondary/AI/translation details.
- Use Bima mascot to make the app feel human.
- Keep cards rounded and soft.
- Build real components from visual references.
- Preserve handmade scrapbook identity.

DO NOT:
- Use the reference page as a single background image.
- Use baked text from images.
- Build unreadable tiny labels.
- Overuse stickers until the UI becomes noisy.
- Make the AI feel hidden or autonomous.
- Make report flow visually scary.
- Use casino, odds, cash, betting, or gambling visuals.
- Make the app look like a political news app.
- Replace functional controls with decorative images.
