OPPOSE SPRINT 12 REAL APP BACKEND PLANNING
Date: 2026-05-19

Purpose:
This document plans the transition from the current frontend/mock-state Oppose MVP into a real backend-backed application. The first production target is Oppose. The backend should be designed so some foundations can later support Clash, but Sprint 12 should not build Clash features inside Oppose.

Current status:
- Oppose Flutter frontend MVP exists through Sprint 11.
- MVP routes, onboarding, home, chats, room setup, live room UI, AI drawer, summaries, safety/reporting, profile, and social graph are implemented with mock/local state.
- flutter analyze passes.
- flutter test passes with 23 widget tests.
- Android debug APK build is blocked by a malformed local NDK install, not app source errors.

Core Sprint 12 goal:
Define and prepare the real backend architecture, data model, API boundaries, realtime plan, AI orchestration plan, moderation plan, and migration path from local mock state to production services.

Sprint 12 should be a backend foundation planning and scaffolding sprint, not a full backend implementation sprint.

================================================================================
1. PRODUCT PRINCIPLES FOR REAL BACKEND
================================================================================

Oppose must remain:
- friend-first
- consent-first for AI
- safe for respectful disagreement
- warm and social, not corporate
- transparent about AI behavior
- explicit about what is saved, shared, summarized, or reported

Backend systems must enforce:
- user identity and session security
- explicit AI consent state
- room membership and friend-only access rules
- private vs shared summary visibility
- report privacy
- block/mute effects
- no hidden AI listening
- no real-money, betting, gambling, crypto, or token economy

Backend systems must not assume:
- every room is public
- every room should be recorded
- every conversation should be summarized
- AI can observe silently without user action
- reports can be ignored because they are mock UI today

================================================================================
2. RECOMMENDED BACKEND DIRECTION
================================================================================

Recommended practical architecture for the next phase:

Mobile app:
- Flutter
- current app remains the client
- replace local controllers gradually with repository/service interfaces

Backend API:
- TypeScript backend using NestJS or Fastify
- REST for normal CRUD
- WebSocket gateway for presence, room state, typing, and lightweight realtime events

Database:
- PostgreSQL
- Prisma ORM or Drizzle ORM
- row-level access checks enforced in service layer

Realtime voice:
- LiveKit Cloud or self-hosted LiveKit for WebRTC/SFU
- backend creates room tokens
- Flutter client joins LiveKit rooms with scoped tokens

AI orchestration:
- backend AI service layer, not direct model calls from Flutter
- model provider can be OpenAI-compatible initially
- all AI calls require explicit user-triggered action or explicit room setting
- AI request logs store metadata, not raw private content unless user consent and retention policy allow it

Object storage:
- S3-compatible storage for avatars and future attachments
- not required for first backend sprint unless profile image upload is prioritized

Notifications:
- Firebase Cloud Messaging later
- not Sprint 12 P0

Analytics/error monitoring:
- Sentry for crashes/errors
- product analytics later, after privacy policy decisions

Why this recommendation:
- PostgreSQL handles social graph, room membership, reports, summaries, and chat well.
- TypeScript backend has strong ecosystem for auth, API, realtime, and AI orchestration.
- LiveKit avoids building WebRTC/SFU infrastructure from scratch.
- Keeping AI calls server-side protects model keys and centralizes consent/safety rules.

Alternative backend options:

Option A: Supabase-first
- Pros: fast auth, Postgres, storage, realtime, row-level security.
- Cons: complex realtime voice still needs LiveKit; advanced AI/moderation orchestration may outgrow edge functions.
- Use if speed is more important than custom backend control.

Option B: Firebase-first
- Pros: fast mobile auth, FCM, realtime data, good Flutter support.
- Cons: relational social graph and reporting workflows become less natural; voice and AI still external.
- Use if push/realtime mobile speed is the highest priority.

Option C: Custom TypeScript + PostgreSQL + LiveKit
- Pros: best long-term control, clear service boundaries, good fit for social graph/moderation/AI.
- Cons: more setup than BaaS.
- Recommended for building a real scalable product foundation.

6. FLUTTER MIGRATION STRATEGY
================================================================================

Current Flutter app uses controllers and scopes with local/mock state.

Migration approach:
1. Keep UI widgets mostly intact.
2. Add repository interfaces under lib/services or lib/repositories.
3. Move mock logic behind repository implementations.
4. Add API implementations after backend exists.
5. Swap controllers from direct mock data to repositories.
6. Preserve widget tests with mock repositories.
7. Add integration tests later.

Recommended client layers:
- controllers/scopes: presentation state
- repositories: app operations and data fetching
- api clients: HTTP/WebSocket implementation
- DTOs: network payloads
- domain models: app concepts

First repositories to introduce:
- AuthRepository
- UserRepository
- SocialRepository
- MessagingRepository
- RoomRepository
- AIRepository
- SafetyRepository
- SummaryRepository

Do not connect Flutter directly to AI providers or LiveKit admin APIs.

7. AUTH AND IDENTITY PLAN
================================================================================

Production auth requirements:
- support email/phone or provider auth
- stable server user ID
- username uniqueness
- account creation timestamp
- AI consent versioning
- optional age/region gates later if required

Recommended first implementation:
- use Supabase Auth, Firebase Auth, or custom JWT provider depending on backend choice
- backend validates token on every request
- backend owns app user record

Important:
- do not trust client-provided user IDs
- do not let username changes bypass uniqueness check
- do not let AI features run before AI consent is accepted

================================================================================
8. REALTIME CHAT PLAN
================================================================================

Minimum real chat:
- persistent conversations
- persistent messages
- participants
- read state
- block/mute enforcement

Realtime options:
- WebSocket gateway owned by backend
- managed realtime from Supabase/Firebase if BaaS chosen

Rules:
- blocked users cannot send direct messages to blocker
- muted users can still exist but notifications/audio can be suppressed
- AI messages must be clearly marked sender_type=ai
- system messages must be clearly marked sender_type=system

================================================================================
9. REAL VOICE ROOM PLAN
================================================================================

Recommended: LiveKit.

Backend responsibilities:
- create LiveKit room when Oppose room starts
- generate scoped LiveKit participant tokens
- validate user membership before token generation
- handle LiveKit webhooks for join/leave if needed
- map LiveKit participant identity to Oppose user ID

Flutter responsibilities:
- request microphone permission
- join room with token
- render participants and speaking state from LiveKit events
- let user mute self
- show connection state from LiveKit

Do not build custom WebRTC/SFU in early real-app phase.

================================================================================
10. AI ORCHESTRATION PLAN
================================================================================

AI must be backend-mediated.

AI principles:
- user-triggered by explicit action
- visible in UI
- consent checked server-side
- memory off by default
- no hidden listening
- no autonomous actions

AI request types:
- quick action: summarize so far, suggest better question, explain both sides
- prompt: user asks a question
- translation: only requested text/messages
- room summary: after room ends and summary setting allows it
- moderation assist: internal/admin later, not user-facing autonomous action

Input sources:
- direct prompt text
- selected chat messages if user asks
- room transcript only if transcript capture is explicitly enabled and consented
- room metadata for lightweight summaries if no transcript exists

Retention policy needed before production:
- whether prompts are stored
- whether responses are stored
- whether transcripts are stored
- how deletion works
- how summary deletion works

================================================================================
11. SAFETY AND MODERATION PLAN
================================================================================

Minimum production moderation:
- reports persist to database
- reports include target type, target ID, reason, details, reporter ID
- reports are private to reporter and moderation/admin systems
- block/mute are enforced in chat and room token generation where relevant
- backend records moderation status

Future admin tools:
- report queue
- report detail view
- user history
- room/message context
- action buttons: dismiss, warn, suspend, remove content, escalate

Do not expose report status publicly until policy is defined.

================================================================================
12. SECURITY AND PRIVACY REQUIREMENTS
================================================================================

P0 requirements:
- authenticated requests only for private app data
- server validates ownership/membership
- server validates friend/room access
- server validates block/mute restrictions
- server validates AI consent before AI calls
- model API keys never ship to Flutter client
- LiveKit admin keys never ship to Flutter client
- reports are not readable by normal users other than limited reporter acknowledgement

P1 requirements:
- rate limiting
- abuse detection for reports/messages/invites
- audit logs for moderation actions
- account deletion workflow
- data export/deletion policy
- secure media upload path

17. IMMEDIATE NEXT ACTIONS
================================================================================

Before implementing Sprint 12, decide:
1. Backend stack: custom TypeScript + Postgres + LiveKit, Supabase-first, or Firebase-first.
2. Backend location: Oppose/backend or root backend.
3. ORM: Prisma or Drizzle.
4. Auth provider: Supabase Auth, Firebase Auth, Auth0, Clerk, or custom JWT.
5. Voice provider: LiveKit Cloud or self-hosted LiveKit.
6. AI provider: OpenAI-compatible API first, with provider abstraction.

Recommended default decisions if owner wants fastest high-quality path:
- Backend: custom TypeScript with Fastify or NestJS
- Database: PostgreSQL
- ORM: Prisma
- Voice: LiveKit Cloud first
- Auth: Supabase Auth or Clerk for speed, backend-owned user profiles
- AI: server-side OpenAI-compatible abstraction
- Location: Oppose/backend

================================================================================
18. FINAL RECOMMENDATION
================================================================================

Do not start real AI, real voice, or real chat all at once.

Start with backend skeleton, schema, auth/profile persistence, and API contracts. Then migrate one app surface at a time from mock state to real repositories.

The safest order is:
1. Backend skeleton and database.
2. Auth/profile persistence.
3. Social graph.
4. Direct chat persistence.
5. Room creation/lobby persistence.
6. LiveKit real voice.
7. AI orchestration.
8. Reports/moderation queue.
9. Notifications.

This order turns the demo MVP into a real product without breaking the polished frontend loop already built.
