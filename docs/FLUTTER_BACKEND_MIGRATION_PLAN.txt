OPPOSE FLUTTER BACKEND MIGRATION PLAN
Date: 2026-05-19

Purpose:
Plan how to migrate the existing Flutter mock MVP to real backend services without breaking the polished user flow built through Sprint 11.

Current Flutter architecture:
- UI screens use lightweight InheritedNotifier scopes.
- Controllers hold local/mock state.
- Mock data lives in `lib/state/mock_data/mock_oppose_data.dart`.
- Service interfaces exist for MVP boundaries.
- Sprint 13 adds repository interfaces and mock implementations for auth/profile migration.

Migration rule:
Do not rewrite the UI. Add repository boundaries and move one controller at a time from mock data to backend-backed repositories.

================================================================================
TARGET CLIENT LAYERS
================================================================================

UI widgets:
- Render screens and components.
- Should not call HTTP directly.

Controllers/scopes:
- Own presentation state.
- Call repositories.
- Expose app-friendly state to widgets.

Repositories:
- Own domain operations.
- Hide mock/API implementation details.
- Return domain models or app DTOs.

API clients:
- Own HTTP/WebSocket details.
- Add auth headers.
- Parse network DTOs.

DTO mappers:
- Convert backend DTOs to existing domain models.

================================================================================
MIGRATION ORDER
================================================================================

Step 1: Repository interfaces with mock implementations
- Add repository interfaces. Sprint 13 starts this with `AuthRepository` and `UserRepository`.
- Move current mock controller logic behind mock repositories one controller at a time.
- Keep all UI behavior identical.
- Existing widget tests should still pass.

Step 2: Auth/profile backend integration
- Connect onboarding/profile to backend endpoints from Sprint 13. Sprint 14b adds the backend repository implementation and wiring.
- Persist username, display name, interests, and AI consent through `UserRepository`.
- Add loading/error states. Sprint 14b adds these for username/profile, interests, AI consent, and profile edit saves.
- Keep mock mode default. Backend mode is enabled by dart defines.

Step 3: Social graph backend integration
- Load friends and friend requests from backend.
- Persist block/mute.
- Update room invite selector to use backend friends.

Step 4: Chat persistence
- Load conversations/messages from backend.
- Send messages to backend.
- Add optimistic send state.
- Add block enforcement errors.

Step 5: Room backend persistence
- Create room on backend.
- Persist room settings and invites.
- Join/leave/end room via backend.

Step 6: LiveKit voice integration
- Request backend LiveKit token.
- Join LiveKit room from Flutter.
- Replace mock participant/speaking/connection state with LiveKit events.

Step 7: AI backend integration
- Send AI quick actions/prompts to backend.
- Use server-side consent checks.
- Keep Memory Off visible unless backend policy changes.

Step 8: Summary backend integration
- Generate/fetch summaries from backend.
- Save/share/delete via backend.

Step 9: Safety backend integration
- Submit reports to backend.
- Persist block/mute.
- Add report confirmation IDs.

================================================================================
ERROR AND LOADING STATES
================================================================================

Every backend-backed controller needs:
- isLoading
- errorMessage
- retry action where useful
- optimistic update rollback where useful
- safe empty state

Do not block the whole app for one failed surface.

Recommended user copy:
- Network unavailable: "Could not connect. Try again."
- Auth expired: "Please sign in again."
- Blocked action: "This action is unavailable because this user is blocked."
- AI consent missing: "AI needs your consent before it can help."
- Report failure: "Report could not be submitted. Please try again."

================================================================================
TESTING STRATEGY
================================================================================

Keep current widget tests.

Add repository tests:
- mock repository returns expected domain objects
- API mappers parse DTOs correctly
- controllers handle success/loading/error states

Sprint 13 added repository tests for mock auth and user repositories.

Sprint 14b added repository tests for backend user repository mapping, request payloads, dev-auth headers, and structured backend errors.

Add backend contract tests later:
- health endpoint
- auth guard behavior
- username availability
- message send
- room create
- report submit

Add integration tests later only after backend APIs stabilize.

Backend mode run flags:
```bash
flutter run \
  --dart-define=OPPOSE_USE_BACKEND=true \
  --dart-define=OPPOSE_BACKEND_URL=http://127.0.0.1:4000 \
  --dart-define=OPPOSE_DEV_USER_ID=00000000-0000-4000-8000-000000000001
```

For Android Emulator, use `http://10.0.2.2:4000` for `OPPOSE_BACKEND_URL`.
